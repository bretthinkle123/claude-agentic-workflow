# Data-lifecycle conventions (retention → erasure → subject rights)

> **On-demand.** Invoke from **planning** (declare retention + deletion path per stored field,
> emit `data_lifecycle` criteria; makes DSR features acceptance criteria instead of afterthoughts),
> **implementation** (build erasure through **one lifecycle facade** so the cascade is complete),
> and **security** (check the erasure cascade actually covers every copy the diff creates).
> Complements `data-protection-conventions`: that skill protects data **while stored**; this one
> governs **how long** it is stored and **how it dies**.

## The rule this enforces

> Every stored personal/sensitive field has a **declared retention period + deletion path**;
> user-initiated erasure **cascades to every copy** (or records why a copy is exempt);
> data-subject-rights flows (**export**, **delete-account**) are planned as product features with
> tests — never bolted on after a regulator asks.

**Honesty:** criteria-gated (`data_lifecycle` rows ride `criteria_covered`), not
surface-reconciled — an undeclared copy (a cache nobody mentioned) escapes the cascade; the
security agent's diff review of data sinks is the backstop. Same accountability-not-proof bar as
the rest of the data controls.

## What planning declares (Data section of plan.md)

Extend the `data-protection-conventions` classification table with two columns:

```
| field | class | at-rest mechanism | retention | deletion path |
| users.email | personal | SSE | life of account + 30d | hard delete on account deletion |
| orders.* | personal | SSE | 7y (tax law) — survives account deletion | anonymize user linkage |
| audit_events.* | personal (actor ids + origin IP, values-free) | SSE | 6y (HIPAA) | exempt from erasure (legal-obligation basis) |
```

Every retention that **outlives account deletion needs a stated legal basis** (tax, audit trail,
fraud) — "we might want it" is not one; that's the data-minimization failure GDPR fines.

## Deletion semantics

- **User-initiated erasure ⇒ hard delete** (or true anonymization — irreversible unlinking, not a
  `deleted_at` flag). **Soft delete is a product feature** (undo, recovery window), not an erasure
  mechanism; if the plan wants one, it gets a **purge job** with a declared window (e.g. 30 days)
  and the erasure clock includes it.
- **The cascade is the hard part** — erasure covers **every copy**: primary rows, FK'd children,
  caches (Redis TTLs count only if ≤ the erasure window), search indexes, analytics/warehouse
  copies, object storage (uploads, avatars), third-party processors (support desk, email tool —
  call their deletion APIs or record the gap for the human's DPA list).
- **Backups honesty rule:** point-in-time backups are **not** scrubbed per-user — that's
  industry-accepted, but it must be **declared, not silent**: backup retention ≤ N days, so erased
  data ages out of all backups within the erasure window + N; a restore replays deletions or
  re-runs the purge job. Logs need no scrub if `logging-conventions` redaction held (PII was
  hashed/opaque from the start).
- **Crypto-erasure** (the elegant path at scale): encrypt each user's sensitive fields under a
  **per-user data key** (`data-protection-conventions` KMS envelope) — destroy the key ⇒ every
  copy, including backups, is ciphertext with no key. Strongest guarantee; plan it when
  sensitive-PII fields already require field encryption anyway.

## Data-subject-rights product features (GDPR/CCPA)

- **Export-my-data**: machine-readable (JSON), covers every personal field of the requester and
  nothing of anyone else's, gated by fresh re-auth (`auth-patterns` — an export endpoint is an
  exfiltration primitive), rate-limited, and audit-trailed (`audit-trail-conventions`).
- **Delete-my-account**: in-product (GDPR/CCPA expect it; Apple **requires** it for App Store apps
  with accounts — `app-store-submission-requirements` already covers the store side), fresh
  re-auth, runs the full cascade above, leaves the audit trail + legal-basis rows, confirms to the
  user. Statutory clocks (GDPR ~30 days, CCPA 45) bound the purge-job window.

## What the test proves

Per `data_lifecycle` criterion (shape in `test-conventions`): seed a user with data in **every
declared copy** → run the erasure flow → **query each raw store** and assert the data is gone or
anonymized, exempt rows (audit events, legal-basis rows) survive, and the erasure itself left an
audit event. For export: the payload contains the requester's declared fields and **no other
user's data** (the cross-owner shape applied to DSR).
