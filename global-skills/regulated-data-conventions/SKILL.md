# Regulated-data conventions (regime → controls entry point)

> **On-demand.** Invoke from **planning** when the feature touches regulated data. This skill is a
> **router, not a control**: it maps the regime to the engineering controls the pipeline can build
> and enforce, names the skill that owns each one, and tells you what to hand back to the human.
> The controls themselves live in the owning skills — invoke those next.

## The rule this enforces

> When a brief involves regulated data, the plan **names the regime**, invokes the owning skill for
> each triggered control, and **surfaces the organizational gaps to the human** — the pipeline
> never claims an app is "compliant" (compliance attaches to the organization and its processes),
> only that **named engineering controls are built and accountable**.

## Regime → control map

For each regime that applies, the rows are the controls to plan. **Owning skill** is what you
invoke; most are already mandatory pipeline-wide — the regime just raises the stakes.

| Regime | Scope trigger | Engineering controls it demands | Owning skill / mechanism |
|---|---|---|---|
| **HIPAA** (US health) | PHI: health status, diagnoses, treatment, health-adjacent free text | Encrypt PHI in transit + at rest (classify PHI **sensitive-pii** → KMS field-encryption); unique user identity + MFA; **per-record access trail including reads**; retention ≥ 6 years for the trail; session timeout | `data-protection-conventions`, `auth-patterns`, `audit-trail-conventions`, `data-lifecycle-conventions` |
| **SOC 2** (B2B SaaS attestation) | Customer data generally | Access control + MFA; encryption; change management **with evidence**; monitoring/alerting; incident response hooks; logical tenant isolation | `auth-patterns`, `data-protection-conventions`, `ci-conventions` + `delivery-conventions` (the PR/CI/SBOM/provenance chain **is** the change-management evidence), `observability-conventions`, `audit-trail-conventions` |
| **PCI DSS** (card payments) | PAN, CVV, expiry, cardholder name | **Never store CVV — ever, encrypted or not.** Don't touch PAN at all: tokenize via the processor (Stripe/Braintree Elements-style, card data never transits your servers) → most of PCI collapses to SAQ-A. If PAN must be stored (rare; challenge it at the plan checkpoint): field-encrypt, mask on display (first 6/last 4), key-custody separation | processor tokenization (plan-level decision), `data-protection-conventions`, `audit-trail-conventions` |
| **GDPR / CCPA-CPRA** (EU / California personal data) | Any personal data of EU/CA subjects | Data minimization (challenge every collected field); **retention limits + erasure paths**; **data-subject-rights product features** (export-my-data, delete-my-account); consent/lawful-basis recorded where required; breach-detection hooks (the 72-hour clock needs alerting to start it) | `data-lifecycle-conventions`, `data-protection-conventions`, `observability-conventions` |

Regimes stack: a US health app sold B2B triggers HIPAA **and** SOC 2 — take the union of rows.
When no listed regime fits but the data is still regulated (FERPA, GLBA, regional laws), map by
analogy — the same four control families (protect, gate, trail, lifecycle) cover them.

## What planning emits (the contract)

1. A **`## Compliance target`** line in `plan.md` naming the regime(s) and the triggering data —
   or explicitly `none` when user data is present but unregulated (silent absence is what audits
   flag; a considered `none` is fine).
2. The **triggered controls as typed acceptance criteria** via the owning skills —
   `data_protection` rows (classification), `audit_trail` rows, `data_lifecycle` rows. They ride
   the existing `criteria_covered` machinery; this skill adds **no new artifact**.
3. For PCI: a stated **tokenization decision** (which processor, confirmation PAN/CVV never touch
   app storage) — and a `data_protection_waiver` is **never** acceptable for CVV.

## The organizational honesty list (surface, don't absorb)

These are required by every regime above and **cannot be built by the pipeline**. Put them in
`plan.md` under `## Out of scope — organizational compliance controls (human-owned)` so the plan
checkpoint shows the human what remains theirs:

- BAAs (HIPAA) / DPAs (GDPR) with every vendor touching the data — including cloud + monitoring
- Written policies, employee training, risk assessments, audit engagement (SOC 2 auditor, PCI QSA/SAQ)
- Breach-**notification procedure** (the app provides detection/alerting; humans do notification)
- Access **reviews** (the app provides roles + the trail; periodic review is a human process)

Never let these silently disappear — an app with perfect field encryption and no BAA is still a
HIPAA violation, and the human needs to hear that from the plan, not from an auditor.
