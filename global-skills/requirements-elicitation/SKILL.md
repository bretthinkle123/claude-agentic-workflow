# Requirements elicitation (pre-planning)

The pipeline's most expensive place to discover an ambiguity is a remediation loop; the
cheapest is a conversation before planning. This skill is that conversation. It is
**operator-invoked and human-facing** — you (main thread) interview the operator, they
answer, and you write `.pipeline/requirements.md`. It is **never invoked autonomously** by
the orchestrator, and it writes no code and no plan.

*(Method adapted from mattpocock/skills `grill-me` (MIT) — a relentless interview to
sharpen a plan — reframed to this pipeline's artifacts. See `global-skills/VENDORED.md`.)*

## When to run it

At the very start of a feature, **before** the design-spec stage and planning, when the
brief is a thin one-liner, the feature touches money/auth/PII, or the operator wants the
ambiguity pinned down. Skip it for a small, well-specified change — planning + plan-audit
already handle those. The operator decides; the orchestrator must not launch it.

## The interview

Ask one focused question at a time and keep going until the operator says they are done —
do not stop after two or three. Do not start writing `requirements.md` until you have
pressed on each area below that applies. Prefer concrete either/or questions over open
ones ("when the payment provider times out, do we retry, queue, or fail the request?"
beats "how should errors work?"). Probe at least:

- **Core behavior & scope** — what exactly must this do; what is explicitly NOT in scope.
- **State transitions** — the lifecycle of each entity (draft→active→archived?), what is
  legal, what is forbidden, what is irreversible.
- **Edge & boundary cases** — empty, first-run, maximum, concurrent, duplicate,
  out-of-order, partial-failure.
- **Failure behavior** — for each external dependency and each user error: retry / queue /
  reject / degrade? what does the user see? is it idempotent?
- **Non-functional budgets** — latency/throughput targets, data volume, expected
  concurrency (these become perf acceptance criteria; a number here is worth a dozen
  later). **Unbounded-collection budget (M4″ ledger F4-05):** whenever an endpoint
  returns a collection and the operator declines pagination, do not stop there —
  ask the expected maximum size AND the behavior at excess (hard cap? 413? accepted
  risk, stated?). M4″ pinned "no pagination" and nobody asked what bounds the
  response; the shipped SELECT had no LIMIT of any kind — a DoS lever at scale.
  "No pagination" is a UX answer; "no bound" is a resource decision the operator
  must make explicitly.
- **Data & trust** — what is stored, its sensitivity (drives data-protection controls),
  who may see/act on each resource (authz boundaries).
- **Assumptions to confirm** — anything you are about to assume, state it and get a
  yes/no rather than guessing.

Treat the operator's answers as the source of truth. If an answer contradicts an earlier
one, surface the contradiction and resolve it.

## Output — `.pipeline/requirements.md`

Write a single file with exactly these three sections:

```markdown
# Requirements — <feature>

## Resolved
- <question> → <the operator's decided answer>
- ...

## Open
- <question the operator deferred or couldn't answer yet> — <why it's open>
- ...

## Out of scope
- <thing explicitly excluded, so planning doesn't build it>
- ...
```

Every question you asked lands in exactly one section. An `Open` item is not a failure —
it is an honest flag that planning must either turn into an acceptance criterion with a
sensible default (and say so) or raise at the plan checkpoint for the human. Nothing here
is untrusted-input-shaped (it is the operator's own words), but it is **input, not a
gate** — planning reads it, the deterministic gates never do.

## Handoff to planning

`requirements.md` becomes planning's authoritative brief: planning turns each `Resolved`
item into scope/criteria, each `Open` item into either a defaulted criterion (stated) or a
checkpoint question, and treats `Out of scope` as hard exclusions. Its absence changes
nothing — planning proceeds from the raw brief exactly as it does today.

**Session isolation (M4″-A2).** After writing `requirements.md`, STOP and tell the
operator to kick off the pipeline in a **fresh session** — `requirements.md` is the
sanctioned channel between elicitation and planning, and the interview conversation
should not precede orchestration in the same context window (it biases the
orchestrator's escalation framing even though stage agents spawn with fresh
contexts). You cannot mint a new top-level session yourself, so this is a handoff
instruction, not a mechanism. If the operator kicks off in-session anyway, proceed
(operator's call) but record it as a logged deviation in the run's touchpoint
journal so the auditor sees it.
