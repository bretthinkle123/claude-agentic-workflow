# Observability conventions — operating a deployed app (PR O)

The deployed counterpart to `logging-conventions`. That skill defines *what* the app emits
(structured fields, PII rules, trace-ID propagation, the CloudWatch backend); **this one wires
the operational surfaces that watch a live service** and close the loop back to the pipeline.
It is per-project infra the pipeline scaffolds — no new gate hooks.

## Error tracking — Sentry (release-tagged to the deploy SHA)

- Sentry SDK on backend **and** frontend/mobile. Initialize with `release = <commit SHA>` and
  `environment = <staging|prod>` — the **same SHA** PR M tagged the image with and PR N
  deployed, so every incident ties to an immutable, signed artifact and to a diff in
  `docs/decisions/`.
- **Redact at the SDK boundary**, not after: `beforeSend` strips PII/secrets per the
  `logging-conventions` PII rules — Sentry must not become an unredacted PII store (and the
  triage agent then re-redacts on the way into the brief — defense in depth).
- Sample errors at 100%; sample performance traces (lower) to control cost.

## Traces + metrics — OTel → CloudWatch / X-Ray

The trace IDs `logging-conventions` already propagates get **exported for real** here: OTel SDK
→ ADOT collector → CloudWatch (metrics/logs) + X-Ray (traces). Emit the RED metrics (Rate,
Errors, Duration) per route as the raw material for the SLOs below.

## SLOs + burn-rate alarms (the signal the canary consumes)

- Define an **SLO per critical user journey** (availability + latency), with an error budget.
- **Multi-window burn-rate alarms** (fast-burn 1h + slow-burn 6h) on the budget — the grown-up
  form of the three minimum alarms `iac-conventions` provisions for PR N. **PR N's canary soak
  rolls back on these alarms**, so an SLO alarm is a *delivery* dependency, not just a
  dashboard: name them so `deploy.yml`'s `<PROD_ALARM_NAMES>` references them.
- Page on fast-burn; ticket on slow-burn. Every alarm names a runbook.

## Synthetic monitoring

Scheduled probes (CloudWatch Synthetics canaries) of the **top user journeys** end-to-end from
outside the VPC — the grown-up successor to `post-deploy-check.sh`'s single health ping. A
failing synthetic is a fast-burn signal.

## Mobile crash reporting (native targets)

For an iOS/Android target: Sentry (or Crashlytics) crash reporting, with **dSYM (iOS) /
mapping.txt (Android) upload wired into the release build** so crashes symbolicate. Release-tag
to the store build number. Ties into the STORE workstream's shipped artifact.

## The triage feedback loop (read-only)

A production incident re-enters the pipeline through the **`triage` agent** (see
`triage-conventions`), never through an auto-fix:
- Operator invokes `triage` with one Sentry issue id → it writes `.pipeline/incident-brief.md`
  (facts + quoted-untrusted evidence + a repo-grounded hypothesis + a suggested next step +
  an injection report), and stops.
- A human decides whether to start a normal pipeline run from that brief — planning →
  plan-approval → … → diff-approval → PR. **Nothing auto-deploys; nothing writes to prod.**
- The Sentry MCP token is **read-only**; telemetry is treated as untrusted input. This is the
  one genuinely new pipeline surface in the delivery workstream, kept honest by the same gate
  discipline as everything else.

## Honest scope

Observability tells you what a deployed system is doing and feeds the auto-rollback + the
triage loop; it does not *prevent* defects (that is the pre-merge pipeline) and it does not
replace human judgment on an incident (triage *summarizes*, humans *decide*).
