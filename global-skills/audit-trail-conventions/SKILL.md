# Audit-trail conventions (who touched which record)

> **On-demand.** Invoke from **planning** (decide which actions need events, emit an `audit_trail`
> criterion per action class), **implementation** (record events through **one audit facade** into
> an **append-only sink**), and **security** (spot-check that the trail calls in the diff cover the
> declared actions). Distinct from `logging-conventions`: app logs answer *"what did the system
> do"* (debugging, redacted, rotated); the audit trail answers *"who touched which record"*
> (accountability, per-record, retained for years). One cannot substitute for the other.

## The rule this enforces

> Every **covered action** on a regulated record produces an **immutable audit event** naming an
> authenticated actor — recorded through **one facade**, stored **append-only**, containing
> **references, never values** — and a test proves the event is written and cannot be altered
> through the app.

**Honesty:** this is criteria-gated (`audit_trail` rows ride `criteria_covered`), **not**
surface-reconciled — there is no deterministic `audit_surface` deploy-gate floor (yet; the
`data_surface` pattern is the template if one is ever warranted). A covered action planning never
declares slips through; the plan checkpoint + security's diff review are the backstop.

## Covered actions (what must produce an event)

- **Reads of sensitive-PII-class records** (the `data-protection-conventions` classification) —
  HIPAA's core demand; a SELECT is an access. Personal-class bulk reads (exports, admin list
  views) too; single-record personal-class reads are a planning judgment call — decide, don't drift.
- **All writes** to regulated records — create, update, delete (including soft-delete/restore).
- **Auth events** — login success/failure, MFA outcome, logout, role/permission change.
- **Exports and erasures** — any `data-lifecycle-conventions` flow (who exported whose data).
- **Admin/support access** — the break-glass path is the one auditors care about most.

## The event schema

```
{event_id, occurred_at (UTC), actor_id, actor_role, on_behalf_of?,   ← WHO (authenticated principal, never "system" for a user-initiated action)
 action, resource_type, resource_id,                                 ← WHAT (opaque ids — NEVER field values: no PHI, no PII, no before/after diffs)
 outcome (success|denied|error), origin_ip, request_id/trace_id}     ← CONTEXT (trace_id per logging-conventions → correlates trail ↔ app logs)
```

The **references-not-values** rule is what lets the trail be retained for years without itself
becoming a sensitive-PII store needing field encryption. If a change needs before/after forensics,
store a pointer to a versioned record, not the data in the event.

## The audit facade (one module, like crypto)

All events go through **one facade** (`audit/` — e.g. `audit.record(actor, action, resource,
outcome)`), called from the service layer, **never** ad-hoc inserts scattered across handlers.
Same rationale as the `data-protection-conventions` crypto facade: coverage becomes a **greppable
named call**, so a covered action missing its `audit.record` is visible in the diff.

**Sink — append-only by construction, not by convention:** the app's DB role on the audit table
has **INSERT/SELECT only** (no UPDATE/DELETE grants) — the facade cannot alter history even when
compromised at the app layer. Managed alternatives: CloudWatch Logs (immutable retention) or
DynamoDB + S3 Object Lock archive. Retention per regime (HIPAA: **6 years**) is declared in
`infra/`, not left to a default. Failure mode is a planning decision per action class: **fail
closed** for regulated writes (no event ⇒ no action, same transaction); reads may fail open with
an alarm — say which in the plan.

## What planning emits

```
AC-AT<n> | audit_trail | <action class, e.g. "reads+writes of patients.* records"> →
          audit event {actor, action, resource, outcome} via audit facade, append-only sink,
          retention=<period> | <service layer / audit/> | audit-event + append-only test
```

## What the test proves

Per `audit_trail` criterion (shape in `test-conventions`): perform the covered action as an
authenticated user → **query the raw sink** → the event exists with the right actor/action/
resource/outcome and **no field values**; then attempt to UPDATE/DELETE that event through every
app-exposed path (and as the app's DB role) → **refused**. A trail the app can rewrite is
decoration, not evidence.
