# Data-protection conventions (classification → at-rest control)

> **On-demand.** Invoke from **planning** (classify every stored field, emit a `data_protection`
> criterion per sensitive field), **implementation** (build the control through the crypto facade),
> and **security** (reconcile the implemented storage surface against the declared class). Makes the
> existing controls — Checkov at-rest, `logging-conventions` redaction, `secrets-management`,
> `auth-patterns`, ASVS 6g — **accountable per field** rather than only advisory or infra-level.

## The rule this enforces

> Every stored field is **classified**; each **sensitive** field has a **declared at-rest
> mechanism** (or an explicit, reasoned **waiver**); a **test** proves the persisted form is not
> plaintext; and the **implemented** storage surface cannot drift from the **declared**
> classification without blocking the deploy.

This is **accountability, not proof** — no static gate can prove "all sensitive data is encrypted"
(the app can always write plaintext somewhere unseen). The teeth are: a material plan-audit flag, a
persisted-form test per field, and a deterministic `data_surface.unprotected` deploy-gate floor.
A **misclassified** field passes silently — human diff-review + the Semgrep/Checkov nets are the backstop.

## Classification taxonomy — the class → control map

| Class | Examples | Required at-rest control | In transit | In logs |
|---|---|---|---|---|
| **Credential / secret** | password, API key, OAuth/session token, secret | password → **slow KDF** (delegate to the auth provider — Firebase default — else argon2/bcrypt/scrypt/PBKDF2, ASVS 11.4.2); other secrets → **not stored** (fetch at runtime via `secrets-management`) or field-encrypted | TLS | **never** logged |
| **Sensitive / regulated PII** | SSN, health, financial account, precise geolocation, biometrics | **field-level (application-layer) encryption** — envelope encryption via **AWS KMS** (per-record/table data key wrapped by a CMK), **on top of** disk SSE | TLS 1.2/1.3 | never logged; hashed/opaque ref only |
| **Personal data** | email, name, phone, address | **SSE at rest** (infra, Checkov-forced) + field-encryption *option* at L3 | TLS | hashed/opaque (`logging-conventions`) |
| **Non-sensitive** | public content, non-PII config | none required | TLS (default) | fine to log |

**Classification tips:** classify by the *worst* interpretation of a field (a "notes" free-text
column that can hold health info is sensitive PII). A pseudonymous analytics id with no linkage to a
person is non-sensitive — but that reasoning belongs in a **waiver**, not a silent pass.

## The crypto-facade rule (non-negotiable)

Route **all** encryption / decryption / hashing / key access through **one module** (e.g.
`crypto/` or a `DataProtection` facade) — never inline crypto scattered across models. New sensitive
fields plug into the facade; they never call a cipher/KDF directly. This is the `code-standards`
facade rule applied to crypto, and it is what makes the security reconciliation (Layer 5) tractable:
the mechanism for a field is a **named facade call**, greppable in the diff.

- **Password KDF:** prefer the auth provider (Firebase/Cognito never expose the raw hash). If the app
  hashes its own, the facade calls a slow KDF (argon2id default) — never a fast hash (md5/sha*).
- **KMS envelope encryption:** the facade requests a data key from **AWS KMS** (`GenerateDataKey`),
  encrypts the field with it **using an AEAD mode — AES-256-GCM default (unique nonce per
  encryption; or use the AWS Encryption SDK, which enforces this), never an unauthenticated mode
  (CBC/CTR/ECB without a MAC)** — stores `{ciphertext, wrapped_data_key, key_id}`, and decrypts on read.
  Key access follows `secrets-management` (runtime fetch, least-privilege IAM on the CMK, no key in
  the tree). Rotation is KMS's / `secrets-management`'s job — name it, don't reimplement it.
- **SSE:** for personal data, declare storage-service SSE in `infra/` (Checkov enforces it).

## What planning emits (Layer 1)

In `plan.md`'s Data section, record each stored field's **class**. In `acceptance.md`, emit a
`data_protection` criterion per **sensitive** field (credential / sensitive-PII / personal):

```
AC-DP<n> | data_protection | <table.field> class=<credential|sensitive-pii|personal>,
          mechanism=<KDF|KMS-field-encryption|SSE>, persisted form is not plaintext | <file/layer> | persisted-form test
```

…**or** a `data_protection_waiver: <reason>` (e.g. "pseudonymous analytics id, no PII linkage").
These ride the existing `criteria_covered` machinery — no new artifact.

## What the persisted-form test proves (Layer 4)

Per `data_protection` criterion: write the value through the repository, **read the raw stored
column/blob**, and assert it is **not the plaintext** (ciphertext or KDF hash), while the app's normal
read still round-trips (decrypt/verify works). For passwords: the stored column is a KDF hash and
`verify(password)` succeeds but the plaintext never appears. See `test-conventions` for the shape.

## What security reconciles (Layer 5)

Security enumerates the **implemented** storage surface (ORM models/columns, file writes, cache
entries, exported blobs), and for each field verifies the declared mechanism is actually present in
the diff (a KDF call, a KMS `encrypt`, SSE in `infra/`). Any **sensitive** field it cannot reconcile
to a mechanism-or-waiver goes in `data_surface.unprotected` — a non-empty list flips `status` off
`clean` and **blocks the deploy** (`deployment-gate.sh` floor, mirrored into loop-exit), converting the
old non-exploitable *warning* into a *block*, regardless of exploitability.
