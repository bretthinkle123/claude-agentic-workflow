# Executive summary & verdict

**Overall grade for building app-store-grade software *today*: B / "strong pre-merge scaffolder, not yet a shipping pipeline."** As an *authoring + pre-merge-review engine* this is genuinely good work — well above typical "vibe-coded agent chains." The deterministic-gate philosophy is real: GREEN is decided by `jq` on status files, never LLM judgment; the loop-exit ≡ deploy-gate invariant is machine-proven across 384+ boundary permutations (`tests/suites/loop-exit-invariant.sh`, verified passing); the test stage produces real property/concurrency/migration/perf/mutation tests with adversarial shapes derived from actual shipped bugs; and the honest-accounting backstops (CVE floor, perf-pairing, marker guards, TTY approval, source-marker guard) meaningfully constrain what a misbehaving agent can slip through. The eval harness (168 assertions) all passes on this tree.

But it is **not ready to safely ship public, app-store-grade software today**, and the top things holding it back are:

1. **There is no single coherent installed pipeline right now.** The two newest security layers — ASVS 5.0 verification and universal input-control (validation + rate-limit) enforcement — live on **two separate unmerged, complementary-half branches**, and the published `~/.claude` copy is a *cross-branch mix*: the live `deployment-gate.sh` is from `feat/input-control-enforcement` (it carries the `input_surface` block), while the live `security.md` and orchestration `SKILL.md` are the older `main` versions. **Result: the live `deployment-gate.sh` enforces an `input_surface` block the live loop-exit predicate does not mirror → the harness-proven loop-exit ≡ gate invariant does NOT hold on the bytes that are actually installed.** *(Correction after re-verification: this is currently **latent, not live** — the published security agent, being main's, never emits `input_surface`, so the gate clause never fires and no real run is blocked today. It becomes an active loop-exit-vs-gate divergence the moment the emitter half is published without the matching loop-exit clause.)* The durable problem is release-management: no coherent source of truth, and the invariant harness validates the *repo*, never the *published* bytes. (H1, M3, P5-01)

2. **The pipeline never enforces a coverage floor.** `deployment-gate.sh` checks `status:"pass"` and acceptance-criteria completeness, but **no gate, loop-exit clause, or hook ever compares `coverage.combined.lines/branches` against a threshold** — despite `test-conventions` telling the reader "gate on combined only: lines ≥ `<N>%`." Coverage % is entirely agent-asserted and advisory. A green suite at 20 % line coverage ships. (H3)

3. **The #1 real-world AI-app failure — broken object-level authorization (IDOR/BOLA) — is guided, not forced.** There is no deterministic requirement that every owner-scoped object has an ownership-assertion *test*; the security agent's row-level-security check (`security.md` step 6b) is a grep heuristic that misses ORM-shaped access, and the input-surface reconciliation gate (the one hard, deterministic app-security gate) covers validation + rate-limit accountability, not authorization. (M2)

4. **A known, real-world defect class is still the *live* guidance.** `api-edge-conventions` (repo + published) still says both "rate limiting sits *before* auth" **and** "key by authenticated identity first" — physically impossible together and the exact root cause of the Ledgerly IP-keyed limiter bug (per operator memory). The two-tier fix exists but is stranded on the unmerged input-control branch. Until merged, the pipeline actively guides implementers toward the bug. (H2)

5. **Everything past the PR is absent (by deliberate deferral).** No CI-as-merge-gate, release/rollback, executed migrations, real-environment load, observability wiring, privacy/retention, a11y, or store-compliance. These are documented Track-2 deferrals, not blind spots — but they are exactly the "professional, production-grade, app-store" surface the operator's goal names, so the honest grade for *that* goal is "pre-merge only." (Info cluster / Perspective 6)

None of these are fatal; all are fixable, and the architecture is sound enough to absorb the fixes without breaking its invariants. The fastest credibility win is a **branch-reconciliation + republish pass** (get main, the branches, and `~/.claude` back to one coherent state) plus a **deterministic coverage floor** — two changes that move the trust needle disproportionately.
